# EVOLUTION.md

本文件回答并总结《Homework 2 · Hint 与 Explore Mode》中所要求的若干演化、设计与反思问题。答案主要基于 sysu-se/con-oo-Ageha-yyd (HW1) 和 sysu-se/ood-Ageha-yyd (HW2) 仓库的代码变更与作业说明，结合本人的实现和理解。

---

## 1. 你如何实现提示功能？

**实现思路：**

提示功能（Hint）要求提供某一格当前可能填写的候选数字集合，或者直接给出下一步的推理型提示。我的实现如下：

- **领域对象驱动**：提示相关逻辑完全封装到领域对象中，由 `Sudoku` 对象（或其方法）负责分析当前盘面，计算某单元格可用的候选数。界面组件调用统一接口（如 `getCandidates(row, col)` 或 `applyHint(...)`）。
- **前端交互**：界面层如操作栏（`Actions.svelte`）通过 Svelte store 向 userGrid store 发出“请求提示”动作，由 store 代理调用下层领域方法。
- **候选数集合**：检查待填格子的同行、同列和同宫已存在的数字，将其排除后得出所有合法候选数，返回数组展示。
- **填空提示**：对于只剩一个候选数的格子，可直接作为下一步的“填空提示”。

举例（见 `src/components/Controls/ActionBar/Actions.svelte`）：

```js
function handleHint() {
  if (hintsAvailable) {
    if ($candidates.hasOwnProperty($cursor.x + ',' + $cursor.y)) {
      candidates.clear($cursor);
    }
    userGrid.applyHint($cursor); // 代理调用领域对象的hint逻辑
  }
}
```

**补充：**
- 提示次数限制，通过 Settings 控制（如 `settings.hintsLimited`，界面有供修改最大提示数的交互）。
- 使用情况统计见 GameOver.svelte。

## 2. 你认为提示功能更属于 Sudoku 还是 Game？为什么？

**我的观点：属于 `Sudoku`，但可与 `Game` 协作。**

### 理由：
- Sudoku 是具体规则和盘面约束的承载体，“在当前盘面下给出候选数”是数独解空间的基础计算，与具体的游戏流程（history/undo等）解耦。提示本质上是对当前盘面逻辑合法性的分析，属于 Sudoku 领域。
- Game 负责的是轮次、历史记录、状态流转（如undo/redo、探索分支），它调用 `Sudoku` 的方法获取候选数或推动盘面变更。对于“剩余多少提示次数”这类全局状态，则由 Game 层管理。
- 协作模式：界面或 store 发起 hint，请求可在 Game 统一收口，由其转发给当前 Sudoku，获得分析结果后响应界面。

## 3. 你如何实现探索模式？

**探索模式（Explore）实现如下：**

- **分支试探**：探索触发后，允许用户对某些不能确定（多候选值）的格子主动选择一个分支尝试填写，并进入一个“探索分支”状态。每次探索都单独记录，不直接影响主历史。
- **状态和历史**：
    - 进入探索模式时，当前盘面与状态快照保存。
    - 后续所有填写操作、提示、撤销都在“探索分支”下操作，有独立的撤销栈和 redo 支持。
    - 可以随时抛弃本次探索分支，回到分支点重新开始选别的路径，或在探索成功后将探索分支提交“合并”到主局面。
- **冲突识别**：若探索过程中出现冲突（某分支无解或无可填合法值），支持直接提示探索失败、允许快速回退。
- **UI 提示与分支**：界面允许用户在主/探索分支之间清晰切换，并显示当前为探索状态（如高亮、标签等）。

---

## 4. 主局面与探索局面的关系是什么？

- **主局面**：指用户标准操作下通过 Game/Sudoku 不断演化得到的主线盘面和历史栈。所有非探索模式下用户操作都直接作用于主局面以及其 history 结构。
- **探索局面**：在主局面某个快照点分叉出来的临时状态分支，允许用户大胆尝试——若满意可“提交”合入主线，否则可丢弃返回主局面重新分支。
- **两者关系**：
    - 入口：每次探索都从主局面当前状态快照克隆，互不影响。
    - 提交/放弃：探索成功时可切换为主局面，失败则丢弃本分支历史归主线，主局面不变。
    - history 结构上体现为探索分支拥有单独的（栈）历史，逻辑上是一种主-副关系，分叉而不相互污染。

## 5. 你的 history 结构在本次作业中是否发生了变化？

**有所变化。**

- **HW1（con-oo）**：只有一条主 history，使用线性栈结构（undoStack/redoStack）实现撤销/重做；
- **HW2（ood）**：为支持探索分支探索，每次探索模式下，需维护一个“分支”history（可以简化为一条副线栈），主线和探索分支各自有独立的历史栈。提交/合并分支时，探索分支历史可并入主 history，如果放弃则直接丢弃。

**实现要点**：
- 默认仍采用栈，但通过状态对象或指针区分当前是在主分支还是探索分支；
- 支持多次分支/回退（树形扩展可选，本次作业用一层分支足矣）。

---

## 6. Homework 1 中的哪些设计，在 Homework 2 中暴露出了局限？

- **单一线性历史**：只适用于顺序撤销/重做，不方便支持用户场景中的分支尝试和多路回退；
- **部分方法（如提示）仅在 UI 或 Store 代理层拼装**：HW1 中部分功能是临时性拼接于界面或 Store 中，未能完全沉淀于领域对象，导致演进时需要迁移业务逻辑到领域层。
- **界面与领域耦合**：初始实现中，部分界面直接调 Store 内部数据结构， HW2 要求必须由领域对象暴露标准接口（如 getCandidates/getHint/applyHint 等）。

---

## 7. 如果重做一次 Homework 1，你会如何修改原设计？

- **早期抽象多分支 history 支持**：将 history 设计成可扩展的树状结构或支持独立分支策略；
- **领域逻辑全面沉淀在 Sudoku/Game 对象**：避免候选数判定、探查、提示等逻辑仅驻留于上层 Store 或 UI；
- **接口/适配器隔离**：UI 层只通过 Store Adapter 或领域对象标准方法与业务交互，不应依赖内部状态暴露；
- **更多测试用例**：对分支、探索、提示、各种异常输入等写充分用例提前暴露复杂情况，减少后续迁移成本。

---

**总结：**
本次作业的代码演化主要是把领域逻辑下沉、接口抽象，以及复杂状态（如探索分支）的历史管理。通过 Store-领域对象分层，使代码更便于扩展和演进，有效支撑了主要的新需求。
